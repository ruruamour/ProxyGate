# ProxyGate Constitution

## Core Principles

### I. Runtime Truth First

Operator-facing state MUST be derived from a single backend source of truth. Frontend views may format data, but they MUST NOT reinterpret backend state through hidden arithmetic or conflicting scope rules.

### II. Safe Proxy Operations

Changes to proxy selection, pool management, subscription handling, or credentials MUST preserve existing security boundaries. Subscription secrets and local/private network targets MUST NOT be exposed or expanded by convenience features.

### III. Regression Coverage For Operator Contracts

Changes to operator-visible behavior MUST include automated regression coverage at the package or API level when the behavior can affect routing decisions, pool health interpretation, or operational safety.

### IV. Pragmatic Single-Service Simplicity

ProxyGate remains a single Go service with embedded WebUI assets unless a split architecture is clearly justified. New abstractions SHOULD be added only when they reduce ambiguity or duplicated logic in runtime code.

### V. Observable Failure Modes

Background systems such as pool maintenance, subscription refresh, and sing-box orchestration MUST fail in a diagnosable way. Status surfaces SHOULD report degraded or stopped states explicitly rather than implying readiness.

## Technical Guardrails

- Go package tests and targeted runtime verification are the default quality gates for changes in this repository.
- Runtime behavior for `free_only`, `custom_only`, and mixed modes MUST remain backward-compatible unless an explicit breaking change is approved.
- Documentation generated by repository tooling SHOULD avoid machine-specific absolute paths and other local-environment leakage.

## Development Workflow

- Every non-trivial feature SHOULD have a spec, plan, and task trail under `specs/` when Speckit is in use.
- Implementation SHOULD update the relevant contract or quickstart notes when operator-visible semantics change.
- Before merge or push, changed code SHOULD pass `go test ./...` and any directly relevant live endpoint checks available in the local environment.

## Governance

This constitution governs repository-level engineering decisions for ProxyGate. Changes that conflict with these principles MUST update this document in the same change set or be rejected. Simpler designs are preferred by default, and exceptions require a concrete runtime justification.
